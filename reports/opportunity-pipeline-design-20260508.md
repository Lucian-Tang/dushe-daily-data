# 创业机会挖掘工作流设计方案

> **作者：** Thomas  
> **日期：** 2026-05-08  
> **背景：** Boss 对创业方向有困惑，团队现有产品线（AI 毒舌日报、Project Ember）均偏向 AI+消费产品方向，需搭建一套系统化的机会发现→评估工作流

---

## 0. 设计原则

1. **源头多元化，落地集中化** — 灵感来自四面八方，评估标准统一
2. **先广后深，宁错杀不放过** — 早期用低成本信号快速过滤，后期才上重量级评估
3. **可自动化尽量自动化** — 重复劳动交给 Cron/API，人类专注判断
4. **一年三次复盘节奏** — Q1/Q2/Q3 末各做一次机会库大盘点，淘汰过时方向

## 2. 筛选过滤层

> **原则：** 快速、低成本、宁可错杀不放过  
> **工具：** Notion/Airtable + 自定义过滤规则

### 2.1 第一轮粗筛（48小时内完成）

**通过条件（满足任意2条即可进入第二轮）：**

| 维度 | 绿灯标准 | 黄灯 | 红灯（直接淘汰） |
|------|----------|------|------------------|
| 市场规模 | TAM > 10亿RMB | 1~10亿 | < 1亿 |
| 团队适配 | 核心能力直接覆盖 | 需补充1~2人 | 需重建团队 |
| 政策友好度 | 无政策风险 | 政策模糊 | 明确限制方向 |
| 技术可行性 | 6个月内可出MVP | 需12个月 | 技术不可达 |

**淘汰记录：** 淘汰原因写入 `filtered/eliminated.md`（方便日后复盘）

### 2.2 第二轮细筛（1~2周完成）

使用下面的评分矩阵，对每个进入第二轮的方向打分：

**评分矩阵（每项1~5分，5分最高）：**

| 维度 | 权重 | 评估问题 |
|------|------|----------|
| 市场规模 | 25% | TAM够大吗？增长趋势如何？ |
| 竞争密度 | 20% | 红海还是蓝海？头部集中度？ |
| 技术可行性 | 20% | 我们能做吗？护城河在哪？ |
| 时间窗口 | 15% | 现在进早不早？晚不晚？ |
| 变现路径 | 10% | 怎么赚钱？多久能盈亏平衡？ |
| 团队热情 | 10% | 创始人愿意All in吗？ |

**通过门槛：** 综合分 ≥ 3.5 进入深度评估层

### 3.2 用户访谈框架

**访谈对象：** 至少 10 个目标用户（至少 3 个有付费意愿）

**访谈结构（每次 30 分钟）：**

```
开场（2min）：
  "我在做一件XX的事，想听听你的真实想法，不用客气"

背景了解（5min）：
  - 你现在怎么解决XX问题的？
  - 每个月花多少钱/时间在这方面？
  - 这件事对你有多重要？

痛点深挖（10min）：
  - 现在方案最让你头疼的是什么？
  - 有没有试过其他办法？为什么没坚持？
  - 如果有一个完美解决方案，你愿意花多少钱？

方案验证（10min）：
  - 如果有一个产品能做XX，你会有兴趣吗？
  - 你最看重什么？价格？效果？服务？
  - 有什么建议给做这个产品的人？

收尾（3min）：
  - 还有谁遇到类似问题？能不能推荐？
```

**判断标准：**
- 问题-方案匹配度（Problem-Solution Fit）：访谈中 ≥ 60% 的人有强烈共鸣 → 通过
- 付费意愿：≥ 30% 表示愿意付费 → 可行

### 3.3 MVP 定义标准

每个方向的 MVP 必须回答以下问题：

```
MVP Checkbox：

[ ] 核心假设是什么？（写出来，不要猜测）
[ ] 最小功能集是什么？（砍到只剩一个最能验证假设的功能）
[ ] 用什么形式做？（网页/App/小程序/内部工具/Hackathon原型）
[ ] 多久能出来？（目标：2~4周内可验证）
[ ] 验证指标是什么？（日活/注册数/付费转化/ NPS）
[ ] 失败标准是什么？（什么结果说明这个方向走不通）
[ ] 最低成本是多少？（时间和钱）
```

### 4.2 绿/黄/红灯评级标准

| 评级 | 标准 | 行动 |
|------|------|------|
| 🟢 **绿灯** | 综合评分 ≥ 4.0，市场大、团队适配、政策友好、时间窗口清晰 | 优先推进，1个月内出MVP |
| 🟡 **黄灯** | 综合评分 3.0~3.9，有明显机会但有1~2个风险点 | 针对性验证风险点，2个月内出结论 |
| 🔴 **红灯** | 综合评分 < 3.0，或任何一维度为红灯 | 暂时搁置，放入观察库，半年后重评 |

**否决项（直接红灯，无论总分多少）：**
- 政策明确限制的方向
- 技术完全不可行
- 市场规模 < 1亿 TAM
- 核心团队能力完全不匹配

## 5. 工具链

### 5.1 信息聚合源 & 自动采集

| 信息类型 | 来源 | 采集方式 | 存储位置 |
|----------|------|----------|----------|
| 融资新闻 | Crunchbase API / IT桔子 | Cron 日 | `raw/financing/` |
| 市场报告 | 官网手动 + RSS | 实时 | `raw/reports/` |
| 社交痛点 | 微博/知乎/V2EX API | Cron 日/周 | `raw/social_signals/` |
| 技术趋势 | GitHub Trending API | Cron 日 | `raw/tech_trends/` |
| 政策文件 | 国务院/发改委 RSS | Cron 日 | `raw/policy/` |
| 海外模式 | PH/IH/YC | 周手动 | `raw/overseas/` |

**Cron 调度建议：**
```
# 每日 08:00 - 融资动态 + 社交痛点扫描
# 每日 09:00 - GitHub Trending
# 每周一 - 周市场报告汇总 + 政策扫描
# 每月末 - 全量数据复盘，生成机会库月报
```

### 5.2 筛选规则（可编程过滤）

在 Airtable/Notion 中设置以下**自动过滤规则**：

```
触发条件（任意满足）：
  [ ] 同一赛道 30天内有 >2 起新融资 → 进入观察
  [ ] 出现新政策支持关键词（AI、新能源、医疗...）→ 进入观察
  [ ] GitHub Trending 出现相关技术 >500 star → 进入观察
  [ ] 社交平台出现相关痛点关键词 >100次/周 → 进入观察

直接过滤条件：
  [ ] 政策明确限制类 → 标记"不投"
  [ ] TAM < 1亿 → 标记"市场规模不足"
  [ ] 技术成熟度 < 3年 → 标记"技术风险过高"
```

## 6. 运作节奏

```
DAILY（日）
  08:00 自动跑融资扫描 + 社交痛点关键词
  09:00 自动跑 GitHub Trending 摘要
  若发现强信号 → 写入 Signals 库，标记"待处理"

WEEKLY（周一下午）
  30分钟机会库review
  - 新增信号 → 快速粗筛（48小时内出结论）
  - 黄灯项目 → 跟踪风险点验证进展
  - 绿灯项目 → 推进竞品拆解 + 用户访谈

MONTHLY（月末周六）
  2小时深度复盘
  - 淘汰库复盘（有没有被错杀的？）
  - 机会库排名变化分析
  - 下月重点关注方向（最多3个）

QUARTERLY（季度末）
  战略级盘点
  - 大盘趋势回顾（政策/技术/资本）
  - 机会库全面重评
  - 确定下季度主攻方向（最多2个）
```
