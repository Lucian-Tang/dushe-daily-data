## 一、产品定位

### 1.1 一句话定义

> **一款挂在耳边/放在桌上的边缘AI设备，实时听你的销售对话，实时给你提示、帮你成单。**

### 1.2 这到底是什么？

**「口袋里的销售冠军教练」**

不是录音转文字的工具（那是Otter），不是会后分析的平台（那是Gong），而是**实时在对话中给你帮忙的AI教练**。

```
用户场景：
销售：「王总，这个价格我们真的给不到...」
  ↓ 实时听
AI耳返：「你刚说了'给不到'，建议换成'这个预算确实有限，我们可以看看其他方案'，避免进入僵局」
  ↓ 
用户：「王总，这个预算确实有限，我们可以看看其他方案...」
```

### 1.3 目标用户与痛点

| 用户 | 核心痛点 | 当前解决方案 |
|------|---------|------------|
| **房产中介** | 带看时客户犹豫，不知怎么逼单 | 老销售带教，但没法随时有师傅 |
| **保险销售** | 客户拒绝不知道怎么接 | 背话术，但实际场景话术不够用 |
| **B2B大客户销售** | 价格谈判、竞品对比时卡壳 | 事后复盘，但当场救不了火 |
| **汽车销售** | 客户犹豫竞品时不知道怎么比较 | 靠经验，没有实时辅助 |
| **4S店/门店** | 新人上手慢，老员工流失后经验带走 | 培训周期长 |

**核心痛点：**
- 销售是在"实时压力"下工作，事后复盘太晚
- 最好的销售经验存在老销售脑子里，没法复制
- Gong/Otter都是"事后分析"，没有"当场救火"

### 1.4 竞品分析

| 产品 | 模式 | 缺陷 | 价格 |
|------|------|------|------|
| **Gong** | 会后分析 + 团队可见 | 实时性为零；贵（$1,500+/用户/年） | 高 |
| **Otter** | 实时转录 + 手动回顾 | 无主动建议；为会议记录设计 | 中 |
| **Chorus** | 对话分析 + CRM同步 | 同样是事后 | 高 |
| **Salesforce Einstein** | AI辅助销售 | 在CRM里，不在对话里 | 很高 |
| **国内：讯飞听见** | 录音转写 | 无建议功能 | 低 |
| **国内：智齿科技** | 客服AI | 场景不同（售后非售前） | 中 |

**市场空白：** 没有一款真正做"实时销售教练"的产品，尤其是面向中国中小sales团队。

## 二、技术可行性

### 2.1 硬件可行性

**PicoClaw 级别硬件足够吗？**

| 参数 | 数据 |
|------|------|
| RPi Zero 2W | 4核ARM @ 1GHz，512MB RAM |
| 成本 | ~$15（单板）+ $5（外壳）= $20/台 |
| 功耗 | 0.5A @ 5V = 2.5W |
| 体积 | 65mm x 30mm，可做成挂耳设备 |

**评估结论：✅ 硬件可行**

但需要注意：
- 实时语音转文字需要模型，PicoClaw 本身不做ASR，需要外接语音服务（Whisper API / 阿里ASR / 讯飞）
- **低延迟挑战**：Whisper本地跑太慢，建议用流式API（如阿里云语音转写，按帧流式返回）

### 2.2 技术架构

```
[麦克风/耳机] → [边缘节点(PicoClaw)] → [流式语音API] → [LLM分析] → [提示返回]
                                    ↓                    ↓
                              [本地缓存/         [CRM同步]
                               离线支持]
                                    ↓
                              [手机App/耳机]
```

**关键技术栈：**
- **ASR（语音转文字）：** 阿里云实时语音转写 / 讯飞实时转写（国内合规）+ Whisper（海外）
- **LLM：** DeepSeek API（便宜 + 低延迟）+ 本地缓存
- **TTS（语音反馈）：** 阿里云TTS / MiniMax（可选耳机播报或手机App推送）
- **CRM集成：** DenchClaw框架（已评估）

### 2.3 延迟挑战

| 环节 | 延迟目标 | 实际预期 |
|------|---------|---------|
| 语音拾取 → ASR | <500ms | 300-500ms（流式API） |
| ASR → LLM分析 | <1s | 800ms-1.5s（DeepSeek） |
| LLM → 用户收到 | <200ms | 500ms（手机App推送） |
| **总计** | **<2s** | **1.5-2.5s（勉强可接受）** |

**实时对话的"2秒法则"：** 超过2秒的反馈会打断对话节奏。  
建议：只做"关键提示"（如对方情绪变化、话题偏离、竞品出现），不做全程实时转译。

### 2.4 技术风险

| 风险 | 缓解方案 |
|------|---------|
| 语音API延迟高 | 本地做VAD（语音活动检测），只传有效语音段 |
| 网络不稳定（带看时4S店信号差） | 本地缓存 + 降级模式（离线记录，事后补分析） |
| ASR准确率（噪音环境） | 建议设备定向麦克风 + 环境噪音检测 |

- 个人付费意愿相对高

**第三波：B2B大客户销售**
- 客单价高，coach价值大
- 但场景分散，标准化难

## 四、MVP 定义

### 4.1 MVP 功能清单（Phase 1，3个月）

**必须有的：**
1. 设备端VAD（语音活动检测）+ 流式ASR接入
2. DeepSeek实时分析（情绪识别 + 话术建议 + 竞品检测）
3. 手机App接收提示（文字 + 可选语音）
4. CRM基础同步（DenchClaw框架，Pipedrive/Salesforce可选）
5. 基础话术库管理（场景：价格谈判/竞品对比/逼单）

**不做的（Phase 2）：**
- 多人会议分析
- 视频分析
- 离线模式
- 团队管理后台

### 4.2 开发周期估算

```
Phase 1（MVP）:
- 硬件选型 + 采购：2周
- 固件开发（PicoClaw集成）：4周
- ASR/TTS对接：2周
- LLM提示逻辑：3周
- App开发（iOS/Android）：4周
- CRM集成：2周
- 测试 + 认证：3周

总计：约10-12周（3个月）

Phase 2（完善）：
- 团队管理后台：6周
- 离线模式：4周
- 多语言/方言支持：4周
```

### 4.3 技术栈复用

| 组件 | 复用度 | 说明 |
|------|--------|------|
| PicoClaw（边缘AI框架） | 100% | 直接用，已有Go版本 |
| DenchClaw（CRM框架） | 100% | 直接用，已适配 |
| Home AI Device架构（RPi5 + Mac Mini + gRPC） | 60% | 可复用的设备管理协议，但RPi Zero是精简版需适配 |
| 语音API（阿里云/讯飞） | 100% | 新集成 |
| DeepSeek API | 100% | 新集成 |
| App（手机端） | 100% | 新开发 |

**复用总结：** PicoClaw + DenchClaw 已有，直接上手。主要是ASR/TTS和App是新工作量。

## 五、风险分析

### 5.1 隐私/合规风险 ⚠️⚠️⚠️（最高风险）

| 风险 | 级别 | 说明 |
|------|------|------|
| **实时录音合规** | 🔴 高 | 中国《个人信息保护法》：录音需对方明确同意。销售场景中"偷偷录音"有法律风险。 |
| **数据安全** | 🔴 高 | 销售对话包含商业机密，数据泄露风险大 |
| **ASR数据留存** | 🟡 中 | 阿里云/讯飞是否留存语音数据需确认 |
| **CRM数据跨境** | 🟡 中 | 若用境外LLM，数据跨境需注意 |

**缓解方案：**
1. 设备端VAD检测"谁在说话"，对方声音不上传（边缘处理）
2. 本地ASR优先（如Whisper本地跑），不上云
3. LLM调用用国内服务商（DeepSeek），数据不出境
4. 明确告知用户"合规使用指南"，并在UI上做确认

### 5.2 硬件供应链风险

| 风险 | 级别 | 说明 |
|------|------|------|
| **RPi Zero 2W断货** | 🟡 中 | 树莓派近年供货不稳 |
| **定制硬件认证** | 🟡 中 | 中国SRRC认证（无线电设备）需要时间 |
| **量产成本** | 🟡 中 | 小批量($20) vs 大批量($8-10)差距大 |

**缓解方案：** 同时验证国产方案（如全志H3 / 瑞芯微RK3308），不做单一供应链绑定。

### 5.3 销售场景多样性风险

| 风险 | 说明 |
|------|------|
| **场景碎片化** | 房产/保险/汽车/ToB销售场景差异大，一套话术库不够 |
| **用户期望管理** | AI不能实时替你说，只能"给你提示"，用户可能觉得不够有用 |
| **"AI教练"接受度** | 销售员不一定愿意被AI"监控"，需要改变心理 |

**缓解方案：** 先做1-2个场景深打（建议汽车4S店 + 保险），打磨话术库，再扩展。

### 5.4 竞品跟进风险

| 风险 | 说明 |
|------|------|
| **Gong/Chorus进入实时赛道** | 可能性中，他们已有语音分析能力，加实时不难 |
| **大厂入局** | 阿里/腾讯/字节做类似产品（有资源 + 有场景） |

**护城河：** 硬件 + 边缘AI + 垂直场景话术库是短期内不易复制的组合。
