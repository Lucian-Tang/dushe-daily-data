## 2. 采集方案

### 2.1 各信源获取方式详解

#### Crunchbase
- **方式：** `crunchbase-api`（第三方封装库，如 `crunchbase4`）或直接调用 Crunchbase REST API
- **可靠性：** ⭐⭐⭐⭐（官方API，数据质量高）
- **限制：** 免费tier有限制（250次/天），付费无限制；部分字段仅付费可见
- **数据量：** 每日可抓 ~200 条融资事件

#### IT桔子
- **方式：** `web_fetch` + 自建爬虫（requests + BeautifulSoup）
- **可靠性：** ⭐⭐⭐（数据以国内早期项目为主，页面结构较稳定）
- **限制：** 大量内容需要登录；反爬较严，建议设置合理间隔
- **数据量：** 每日可抓 ~150 条融资事件

#### 36kr 投融资频道
- **方式：** RSS Feed（`https://36kr.com/feed`）+ `web_fetch` 补充
- **可靠性：** ⭐⭐⭐⭐（官方RSS，更新及时）
- **限制：** RSS仅摘要，详细内容需访问详情页
- **数据量：** 每日 ~50-100 条快讯

#### ProductHunt
- **方式：** `web_fetch`（官方页面 `producthunt.com/posts`，按天分页）
- **可靠性：** ⭐⭐⭐⭐（页面结构稳定，反爬友好）
- **限制：** 需设置 User-Agent，避免高频请求
- **数据量：** 每日 ~20-50 个新产品

#### IndieHackers
- **方式：** `web_fetch`（播客访谈列表 + 产品页面）
- **可靠性：** ⭐⭐⭐（内容需登录才能看完整，高质量内容多在付费区）
- **限制：** 建议仅抓取公开的播客/访谈和产品列表

#### GitHub Trending
- **方式：** `web_fetch`（`github.com/trending`）+ 官方 GraphQL API（需token）
- **可靠性：** ⭐⭐⭐⭐⭐（官方页面，结构最稳定）
- **限制：** 无；但大量抓取建议用GraphQL API
- **数据量：** 每日语言分区各取Top20，约100-200个项目

#### Hugging Face
- **方式：** `web_fetch`（模型列表页 `huggingface.co/models`）+ 公开 API
- **可靠性：** ⭐⭐⭐⭐（官方平台，数据结构化程度高）
- **限制：** 无；部分模型页面可能较大

#### arXiv
- **方式：** RSS Feed（按分类，如 CS.AI/CS.LG）+ `openalex-api`（推荐，更稳定）
- **可靠性：** ⭐⭐⭐⭐⭐（官方RSS，开放访问）
- **限制：** 无；但按需抓取，不建议全量

#### Hacker News
- **方式：** 官方 Firebase API（`hacker-news.firebaseio.com/v0/`）
- **可靠性：** ⭐⭐⭐⭐⭐（最可靠，无反爬，最快）
- **限制：** 无；官方鼓励合理使用

#### Reddit
- **方式：** `praw`（Python Reddit API Wrapper），或 `web_fetch` 作为降级
- **可靠性：** ⭐⭐⭐⭐（API稳定，但有速率限制）
- **限制：** 未认证应用：60请求/分钟；推荐申请专属 App
- **数据量：** 目标子版块每日新帖约50-200条

#### Twitter/X
- **方式：** `snscrape`（无需API，适合历史推文）+ `nitter`（开源替代，无登录）
- **可靠性：** ⭐⭐⭐（页面结构频繁变化，维护成本较高）
- **限制：** 官方API高费用；推荐用 `nitter` 作为降级方案
- **注意：** 建议人工定义关键词列表做定向采集，不做全量

#### V2EX
- **方式：** `web_fetch`（节点列表页 + 帖子列表）
- **可靠性：** ⭐⭐⭐（robots.txt允许，合理频率）
- **限制：** 无；但页面编码需注意（UTF-8）

## 4. 数据处理管道

### 4.1 整体架构

```
[原始数据采集] 
      ↓
[JSON/HTML 原始存储] → 原始层 (raw/)
      ↓
[去重处理] → 去重层 (dedup/)
      ↓
[结构化解析] → 结构化层 (structured/)
      ↓
[分类打标] → 分类层 (classified/)
      ↓
[机会评分] → 评分层 (scored/)
      ↓
[机会卡片生成] → 输出层 (cards/)
      ↓
[推送/存储]
```

### 4.2 去重算法

**目标：** 同一事件/产品在不同信源重复出现，或同一内容的不同版本

**方案：多级去重**

```
Level 1: 精确去重（URL / ID）
  → 用信源自带唯一ID（如 HN item ID、GitHub repo full_name）
  
Level 2: 内容相似度去重（MinHash / SimHash）
  → 适用：融资事件（不同信源报道同一事件）
  → 流程：
    1. 提取：公司名称 + 融资轮次 + 融资金额 + 日期
    2. 规范化：统一货币单位、转小写、去停用词
    3. 计算：Jaccard Similarity > 0.85 → 视为同一事件
    4. 保留：最早出现的信源记录，或信源优先级最高的

Level 3: 语义去重（可选，LLM 辅助）
  → 当 Level 1+2 无法判断时，调用 LLM 判断两条记录是否重复
  → 触发条件：Top-k 候选，且相似度在 0.6-0.85 区间
```

**实现建议：** 使用 `datasketch` 库（MinHash）处理内容级去重，PostgreSQL 存储精确去重键

### 4.3 自动分类

**方案 A：关键词规则（冷启动，快速上线）**

```
分类体系：
  [赛道]  SaaS / AI-ML / 硬件 / 金融科技 / 医疗 / 教育 / 电商 / 工具
  [阶段]  种子 / 天使 / A轮 / B轮及以上
  [类型]  融资 / 产品发布 / 技术突破 / 市场趋势 / 政策
  [地域]  国内 / 海外 / 全球

规则示例：
  IF "AI" OR "LLM" OR "大模型" OR "AGI" in title → 赛道: AI-ML
  IF "Series A" OR "A轮" OR "估值" AND "融资" → 阶段: A轮
  IF "launch" OR "launched" OR "发布" in title → 类型: 产品发布
```

**方案 B：NLP 零样本分类（进阶，快速迭代）**

```
工具：transformers + CLIP / bge-large-zh-v1.5（中文语义 embedding）

流程：
  1. 采集文本 → 拼接：title + description + tags
  2. 向量化 → 用预训练中文模型（bge-large-zh-v1.5）
  3. 零样本分类 → 用 candidate_labels（如 "AI大模型" "SaaS" "硬件"）计算相似度
  4. 取 Top-1 标签 + 置信度分数

优势：无需训练，规则 + 零样本分类互补，覆盖长尾
```

**分类流程建议：**
- Level 1：关键词规则（毫秒级，覆盖 60-70%）
- Level 2：零样本 NLP 兜底（秒级，覆盖剩余 30%）
- Level 3：置信度 < 0.5 的人工复核

### 4.4 机会评分公式

**核心思路：** 信号强度 × 可信度 × 时效性 × 赛道相关性

```
OpportunityScore = S_signal × S_credibility × S_recency × S_relevance

各维度说明：

1. S_signal（信号强度，0-1）
   = f(Engagement, 融资额度, 社区讨论量)
   - GitHub: Star数 > 1000 → +0.3；> 5000 → +0.5
   - HN: 点数 > 100 → +0.3；> 500 → +0.5
   - 融资: 金额 > $1M → +0.2；> $10M → +0.4

2. S_credibility（可信度，0-1）
   = 信源权重 × 数据完整度
   - Crunchbase 官方数据 → 1.0
   - HN/API 数据 → 0.9
   - 爬虫数据 → 0.7-0.8

3. S_recency（时效性，0-1）
   - 24h 内 → 1.0
   - 3天内 → 0.8
   - 7天内 → 0.6
   - 30天内 → 0.3
   - > 30天 → 0.1

4. S_relevance（赛道相关性，0-1）
   = 用户自定义赛道 × 关键词权重匹配度
   - 用户关注 AI → 命中 AI 相关 → 0.9-1.0
   - 用户关注金融 → 命中 fintech → 0.7-1.0

最终分数映射：
  0.8+ → 🔥 重点机会（推送给用户）
  0.6-0.8 → ⭐ 值得关注（进入列表）
  0.4-0.6 → 📋 一般信息（归档）
  < 0.4 → 丢弃或低优先级存储
```

**动态调参：**
- 用户可以对不同赛道配置不同权重
- 可以根据历史点击/阅读数据做 A/B 测试调参

## 5. 交付形态

### 5.1 推荐：飞书多维表格（Bitable）作为主入口

**理由：**
- Boss 和团队日常在飞书，多维表格天然支持筛选、协作、评论
- 每条机会卡片作为一行，支持多维度筛选
- 可以按赛道/类型/评分排序，第一眼抓住重点

**多维表格字段设计：**
| 字段 | 类型 | 说明 |
|------|------|------|
| 标题 | 文本 | 项目/事件名称 |
| 摘要 | 长文本 | 50字描述 |
| 信源 | 单选 | GitHub/HN/36kr/ProductHunt... |
| 链接 | URL | 原始链接 |
| 赛道 | 多选 | AI/SaaS/Fintech... |
| 类型 | 单选 | 融资/产品/技术/趋势 |
| 评分 | 数字 | 0-1 保留两位小数 |
| 发现时间 | 日期 | 自动化填充 |
| 状态 | 单选 | 待研究/已研究/有戏/放弃 |
| 标签 | 多选 | 自定义标签 |
| 讨论区 | 评论 | 团队协作 |

### 5.2 备选：飞书文档（日报/周报）

- 每日自动生成一份「机会日报」，Markdown 格式
- 推送至飞书群或知识库
- 适合高层快速浏览，不需要交互

### 5.3 存储层：本地 / GitHub

- **原始数据 + 处理结果** 存在本地 `data/` 目录，JSONL 格式
- **GitHub 私有仓库** 作为备份和版本控制
- **触发流程：** cron → 采集 → 处理 → 生成卡片 → API 写入飞书多维表格

### 5.4 交付形态总结

| 形态 | 用途 | 更新频率 |
|------|------|----------|
| 飞书多维表格 | 主力工作台，持续积累 | 实时 |
| 飞书文档日报 | 每日推送，管理层阅读 | 每日 |
| GitHub 私有仓库 | 数据备份、代码版本 | 持续 |
