## 1. 信息数据库规模：信源池估算

### 现状评估

当前 5 个信号源（HN / GitHub / arXiv / ProductHunt + 5个日报 cron）主要覆盖 **技术/开发者/创业圈**，存在明显盲区：

| 维度 | 现状 | 缺口 |
|------|------|------|
| 地理覆盖 | 偏英文/硅谷 | 国内政策/监管信息缺失 |
| 内容类型 | 技术为主 | 财经、监管、社会事件不足 |
| 媒体类型 | 科技社区为主 | 传统媒体、专业媒体缺失 |
| 语言 | 英文为主 | 中文信源薄弱 |

### 支撑多源校核的信源池估算

**关键假设：** 单一事件平均需要 3-4 个独立信源交叉验证方可判定可信度等级。

| 信源类型 | 建议数量 | 用途 |
|----------|----------|------|
| 科技/AI 新闻 | 5-8个 | 日常核心覆盖 |
| 开发者社区 | 3-5个 | GitHub/HN/Lobsters/HackerNews |
| 学术/论文 | 2-3个 | arXiv/PubMed/ACL |
| 中文科技媒体 | 3-5个 | 36kr/虎嗅/量子位/机器之心 |
| 创投/财经 | 3-5个 | TechCrunch/我爱码/投资界 |
| 社交舆情 | 2-3个 | Twitter/X 精选账号 |
| 政府/监管公告 | 2-3个 | 官方政策/公告（国内必选） |

**结论：** 建议扩展至 **20-25 个信源**，按优先级分批接入。
- Phase 1（ MVP，3个月）：15 个核心信源
- Phase 2（完善）：扩展至 25+ 信源

## 2. 可信度评分算法

参考 **NewsGuard（5项指标）** 和 **Ground News（媒体偏向+覆盖差异）**，设计适合技术日报场景的评分模型：

### 评分维度（5维 × 20分 = 100分制）

| 维度 | 权重 | 说明 | 评分方式 |
|------|------|------|----------|
| **来源权威性** | 25% | 媒体/账号历史公信力 | 人工维护白名单 + LLM 评估 |
| **事实性** | 25% | 包含原始数据/引用/链接 | NLP 检测是否含原始来源 |
| **交叉验证率** | 20% | N 个信源确认同一事实 | 多源匹配命中数量 |
| **时效性** | 15% | 发布时间距离当前时间 | 时间衰减函数 |
| **独立性** | 15% | 原创报道 vs 转发洗稿 | 文本相似度 vs 首发源 |

### 计算公式

```
总得分 = Σ(维度得分 × 权重) × 交叉验证系数
```

**交叉验证系数（CVF）：**
```
CVF = 1.0 + 0.1 × (确认信源数 - 1)
# 1个信源: 1.0x | 3个信源: 1.2x | 5个信源: 1.4x
```

**星级映射：**
| 总分 | 星级 | 含义 |
|------|------|------|
| 85-100 | ⭐⭐⭐⭐⭐ | 高度可信，事实清晰，多源确认 |
| 65-84 | ⭐⭐⭐⭐ | 较可信，建议参考 |
| 45-64 | ⭐⭐⭐ | 中等可信，独立验证 |
| 25-44 | ⭐⭐ | 存疑，需谨慎 |
| 0-24 | ⭐ | 不可信/低质量/存疑 |

### LLM 辅助评分

使用 LLM 对每条信息做结构化评分（一次 API 调用）：
```
输入：标题 + 摘要 + 全文
输出：JSON {事实性:0-20, 引用质量:0-20, 独立性:0-20, 时效性:0-20, 可行动性:0-20}
```

## 3. 多源交叉验证：技术方案

### 方案选型对比

| 方案 | 精度 | 速度 | 成本 | 适用场景 |
|------|------|------|------|----------|
| 关键词匹配 | 快 | 低 | 高 | 简单事件标题相同 |
| 实体识别（NER）+ 实体链接 | 中 | 中 | 中 | 事件主体相同但表述不同 |
| 向量相似度（Embedding） | 高 | 慢（但可优化） | 高 | 语义相近的报道 |
| 混合方案（推荐） | 高 | 中 | 中 | 生产环境 |

### 推荐方案：三层匹配 pipeline

**第一层：实体指纹（快筛）**
- 用 NER（spaCy/中国本）提取：人名/机构名/产品名/地名
- 生成实体集合指纹，命中即进入候选集

**第二层：语义向量（精准匹配）**
- 用 embedding 模型（text-embedding-3-small 或国产 BGE）将每条信息转为 1536 维向量
- 相似度阈值：>0.82 视为"同一事件的不同报道"

**第三层：LLM 判断（兜底）**
- 对模糊案例让 LLM 判断："这两条信息是否报道同一事件？"

### 事件去重与合并

```
事件簇（Event Cluster）：
  - cluster_id
  - 核心事件描述（LLM 提取的一句话摘要）
  - 参与实体列表
  - 报道列表 [source_1, source_2, ...]
  - 覆盖时间范围
  - 综合可信度评分（加权投票）
```

## 4. 架构选型：完整数据管道

```
[采集层]           [存储层]          [处理层]          [输出层]
   │                │                │                │
   ▼                ▼                ▼                ▼
RSS/API/Cron    →  PostgreSQL      →  Python Worker  → 微信小程序
Webhook         →  (向量库)         →  (交叉验证)     → GitHub Pages
爬虫(Scrapy)    →  Redis(缓存)     →  LLM评分        → API 回调
                →  JSON(MD备份)    →  NER/Embedding  │
```

### 详细流程

**Step 1：采集（Collection）**
- 各信源 cron job（5-30min 间隔）抓取原始内容
- 标准化格式：{id, title, summary, content, source, author, published_at, url}
- 入库 PostgreSQL + 同步写入本地 MD（备份）

**Step 2：清洗（Cleaning）**
- HTML 去除 / 乱码处理 / 标题去重
- 语言检测（可选按需翻译）
- 垃圾信息过滤（低质量标题党检测）

**Step 3：实体提取 & 向量化**
- spaCy/BGE 模型提取实体 + 向量化
- 存入向量数据库（Milvus / Qdrant / pgvector）

**Step 4：多源交叉验证**
- 向量相似度搜索，找到候选事件簇
- 实体指纹匹配，确认同一事件
- 汇总该事件的所有报道

**Step 5：可信度评分**
- LLM 结构化评分（5维度）
- 交叉验证系数计算
- 综合得分 → 星级

**Step 6：生成 Actionable 结论**
- LLM 生成一句话结论："这个发布意味着什么？对你有什么用？"
- 格式：背景 + 要点 + 行动建议（50字内）

**Step 7：输出**
- 日报 JSON → GitHub Pages 静态文件
- 实时 API → 微信小程序拉取
- 异常/重要事件 → 推送通知

### 技术栈建议

| 组件 | 推荐选型 |
|------|----------|
| 后端框架 | FastAPI（轻量 + 自动文档） |
| 数据库 | PostgreSQL + pgvector（向量+结构化合一） |
| 定时任务 | APScheduler / Temporal（生产级工作流） |
| Embedding | text-embedding-3-small（OpenAI）或 BGE（国产） |
| NER | spaCy zh（中文）+ spacy-en |
| LLM 评分 | GPT-4o-mini（成本优先）或 DeepSeek-V3（性价比） |
| 向量库 | pgvector（简单）或 Qdrant（高频） |
| 部署 | Docker + 1台 4C8G 云主机 |

## 5. 成本估算

### 日处理规模假设

| 指标 | 数值 |
|------|------|
| 日处理信息量 | 1,000 条 |
| 涉及交叉验证的信息 | 约 400 条（其余为低价值单源内容） |
| 需要 LLM 评分的信息 | 约 600 条 |
| 需要生成结论的信息 | 约 200 条（Top 20% 高分内容） |

### API 调用量估算（月度）

| 调用类型 | 单条次数 | 日调用 | 月调用（30天） |
|----------|----------|--------|----------------|
| Embedding（采集时） | 1次/条 | 1,000 | 30,000 |

| LLM 结构化评分 | 1次/条 | 600 | 18,000 |
| LLM 生成结论 | 1次/条 | 200 | 6,000 |

## 6. 风险项 & 建议

| 风险 | 影响 | 建议 |
|------|------|------|
| 信源质量参差 | 评分偏差 | 建立信源白名单，动态调整权重 |
| LLM 幻觉导致误判 | 可信度评分失真 | 高风险内容人工复核流程 |
| 中文内容覆盖不足 | 日报实用性降低 | 优先补充 36kr/虎嗅/量子位 RSS |
| 响应延迟（Embedding 计算慢） | 实时性差 | 异步队列 + 预计算 |
| 向量数据库膨胀 | 查询变慢 | 定期清理/归档旧数据 |
