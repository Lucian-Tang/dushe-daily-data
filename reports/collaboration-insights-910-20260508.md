# 团队协作方法论分析报告

## 第1部分：Item 9（角色切换）对我们协作的启示

### 核心洞察

原文指出："同一个AI，产出不同质量的根本原因是没有给它定义清楚场景。"

这句话直接映射到我们的团队协作模式：Boss 的指令经过 Lucia → Thomas/Stephen，这条链路上的每个 agent 如果不先声明自己当前处于什么模式，接收方就无法校准对产出质量的预期。

### 各 Agent 模式定义

#### Lucia（COO/协调者）

| 模式 | 触发条件 | 输出标准 | 质量约束 | 时间约束 |
|------|----------|----------|----------|----------|
| **日常协调模式** | 正常分派任务、进度同步 | 任务清单清晰，责任人和截止时间明确 | 信息无遗漏，转述不失真 | 即时响应（<15min） |
| **紧急问题处理模式** | Boss 直接标记紧急，或线上事故 | 先止血再分析，结论先行 | 容错优先（允许不完整），但方向不能错 | 5分钟内给出初步判断 |
| **审查模式** | Thomas/Stephen 提交产出后 | 对照原始需求逐条核查，标注通过/需修改/存疑 | 严守质量底线，不因时间压力放水 | 按约定交付节点，不卡审 |

#### Stephen（工程师）

| 模式 | 触发条件 | 输出标准 | 质量约束 | 时间约束 |
|------|----------|----------|----------|----------|
| **架构设计模式** | Lucia 分派技术方案设计、技术选型任务 | 方案完整（利弊分析、风险点、依赖项），关键决策有据可查 | 必须符合现有技术栈约束，考虑团队能力边界 | 根据任务规模约定（1-3天） |
| **快速编码模式** | 明确标记"快速原型"、"MVP"、"临时方案" | 功能可用，逻辑基本正确，不要求极致优化 | 可接受技术债，但必须在产出中注明 | 以天为单位，不超过约定deadline |
| **Debug模式** | 线上bug、逻辑错误、测试失败 | 根因定位 + 修复方案 + 回归测试计划 | 修复不能引入新问题，核心路径优先 | 严重bug：1小时内给出临时方案；一般bug：当天内 |
| **报表/文档生成模式** | 生成周报、技术文档、状态更新 | 结构化输出，关键数据准确 | 数据真实，不美化；判断性内容标注置信度 | 按需，即时或约定时间 |

#### Thomas（产品经理）

| 模式 | 触发条件 | 输出标准 | 质量约束 | 时间约束 |
|------|----------|----------|----------|----------|
| **竞品拆解模式** | Boss 或 Lucia 明确要求竞品调研 | 客观还原竞品能力，差异点量化，与我们现状对比清晰 | 不带预设立场，引用来源可查 | 根据范围，1-3天 |
| **PRD写作模式** | 需求确认后的正式PRD输出 | 背景/目标/功能拆解/验收标准完整，逻辑无漏洞 | 每个功能点可测试，避免模糊表述 | 按PRD里程碑交付 |
| **评审模式** | PRD/方案进入评审阶段 | 提前准备评审要点，列出开放问题和备选方案 | 主动推动决策，不把问题留在评审后 | 评审会议前完成材料 |

### Agent Mode Declaration 机制

**每次任务开始，agent 必须显式声明当前模式，格式如下：**

```
[Mode Declaration]
Agent: [Lucia/Stephen/Thomas]
Mode: [当前模式]
Task: [任务描述]
Quality Bar: [高质量优先/速度优先/容错优先]
Time Constraint: [具体截止时间]
```

**为什么重要：**
- **对 Lucia**：知道 Thomas/Stephen 的产出是"速度优先"就不会用"高质量"标准来批判
- **对 Boss**：知道团队各角色当前模式，就能理解为什么某些产出看起来"潦草"但其实是正确的策略选择
- **对 Thomas/Stephen**：有明确的免责边界——当你声明了"快速编码模式"却用"架构设计模式"的质量标准来验收，矛盾前置了

## 第2部分：Item 10（冲突解决）对我们协作的启示

### 核心洞察

原文指出："'谁说了算'不是产品问题，是架构问题。你的设计工具选型其实在定义协作规则。"

对我们而言，Boss→Lucia→Thomas/Stephen 这条链路里，冲突的本质是**权威来源分散**，不是沟通问题。架构问题需要用架构规则来解决，不是靠"多沟通"。

### 冲突场景与解决规则

#### 场景1：Thomas 和 Stephen 对同一任务有不同意见

**典型场景：** Thomas 提出的方案技术成本极高；Stephen 提出的技术方案产品逻辑有漏洞。

**解决规则：**
1. 技术可行性争议 → **Stephen 主导**，Thomas 有举证责任（需证明产品价值值得技术投入）
2. 产品逻辑争议 → **Thomas 主导**，Stephen 有否决权（技术明显不可行的方案不应推进）
3. 若争议持续 → **Lucia 裁决**，裁决依据：谁更接近 Boss 的原始意图
4. **禁止操作**：不允许任何一方单方面宣布方案通过，必须等三方确认或 Lucia 裁决

#### 场景2：Lucia 和 Boss 对优先级有不同判断

**典型场景：** Boss 临时插入任务，Lucia 认为当前 sprint 已满，无法承接。

**解决规则：**
1. Lucia 拥有**当前 Sprint 的执行权**——在 Lucia 认定 Sprint 已满的情况下，Boss 的临时任务必须排队或替换已有任务
2. **Lucia 有义务给出清晰的trade-off分析**：临时任务插入 vs. 哪个已有任务退后，Boss 做最终拍板
3. 紧急情况（Boss 直接标记"紧急"）→ **Boss 优先**，但 Lucia 需在执行后补上对其他任务影响的说明

#### 场景3：多人同时修改同一个文件/飞书文档

**解决规则：**
1. **文件层面**：同一文件同一时间只允许一个 agent 持有写锁
2. **实现方式**：文件名加锁标记 `_LOCKED_by_[agent].ext`，操作完毕后立即释放
3. **飞书文档**：Lucia 独占写权限，Thomas/Stephen 以评论/建议模式参与，Lucia 合并修订
4. **Git 文件**：PR 评审通过前，同一文件的并行修改通过 branch 隔离；合并冲突由最后合并的 agent 负责解决，并通知相关方

#### 场景4：多个 subagent 操作同一个外部资源（GitHub/飞书）

**解决规则：**
1. **Lucia 是唯一有写权限的操作入口**（GitHub PR 合并、飞书文档发布）
2. Thomas/Stephen 的操作（如提交 PR、编辑文档）通过 Lucia 协调，避免并发写入冲突
3. 如需并行操作不同资源（如同时修改两个独立模块），需在任务开始时向 Lucia 报备

### 协作冲突解决规则汇总

| 规则类型 | 具体规则 | 优先级 |
|----------|----------|--------|
| **优先级链条** | Boss > Lucia > Thomas/Stephen（日常执行层 Lucia 有最终解释权） | 最高 |
| **文件冲突预防** | 同一文件同一时间只允许一个 agent 写；用锁文件标记 | 高 |
| **飞书文档编辑权** | Lucia 独占写权限；其他角色用评论模式 | 高 |
| **紧急 Override** | Boss 标记紧急的任务自动升级为最高优先级，覆盖当前 Sprint | 特殊 |
| **争议裁决** | 技术争议→Stephen，产品争议→Thomas，跨域→Lucia | 中 |
| **通知义务** | 任何冲突解决后，相关方必须在合理时间内收到通知 | 中 |

## 第3部分：本周行动计划

### 规则1：启动 Agent Mode Declaration

**规则：** 每次 Lucia 分派任务时，接收方（Thomas/Stephen）必须回复 Mode Declaration，声明当前模式和质量预期。

**受益场景：** 减少"产出质量不符合预期"的返工；Boss 看到模式声明后能理解团队当前负载状态。

**实施方式：**
1. Lucia 在分派任务的消息模板中加入 `[Mode: ?]` 字段
2. Thomas/Stephen 在响应时填写 Mode Declaration（参考上述模板，简化版也可）
3. 在本周的飞书群中试运行，无需额外工具，先用消息模板习惯

---

### 规则2：飞书文档编辑权限收紧

**规则：** Thomas 和 Stephen 在飞书文档中只使用**评论模式**提出修改建议，不直接编辑；Lucia 负责合并修订。

**受益场景：** 避免多人同时编辑导致的内容覆盖；保留修改历史和讨论痕迹；Lucia 掌控最终交付质量。

**实施方式：**
1. Lucia 在相关飞书文档中将 Thomas/Stephen 的权限调整为**可查看+评论**
2. 存量文档中如果已有直接编辑权限，由 Lucia 重新设置权限
3. 本周新建的所有文档继承此权限模式

---

### 规则3：文件操作锁声明制度

**规则：** 当 Stephen 或 Thomas 需要修改一个共享文件（如 `AGENTS.md`、共享配置文件）时，需在飞书群里发一条锁声明（如 `@Lucia 锁定 reports/weekly/plan.md 进行更新，预计完成时间 30min`），Lucia 确认后开始操作，操作完毕立即通知释放。

**受益场景：** 防止并发修改导致的一方改动被覆盖；建立操作的可追溯性。

**实施方式：**
1. 在飞书群公告或群消息中明确此规则
2. 前3天由 Lucia 主动提醒未声明的锁操作，逐步形成习惯
3. 暂不引入技术工具，先用群消息约束；如果频率变高再考虑引入简单的事实记录文件
