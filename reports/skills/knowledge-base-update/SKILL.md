# knowledge-base-update — 知识库归档

## Overview
`knowledge-base-update` 负责识别团队产出中可归档的知识，按分类标签进行结构化归档，更新 `MEMORY.md`（长期记忆）和 `memory/YYYY-MM-DD.md`（日常日志），确保团队的决策、经验和教训不会随时间丢失。

**核心原则：** 凡是经历过一次"为什么"的讨论，结果就应该被归档。不让同一个坑踩两次。

**知识生命周期：**
```
产出 → 识别 → 分类 → 归档到 MEMORY.md + memory/YYYY-MM-DD.md → 飞书同步（可选）
```

## When to Use
- **Daily Report 生成后** — 日报中的决策和教训需要沉淀
- **Skill 设计完成时** — 新 skill 是团队知识资产
- **Blocker 解决后** — 解决方案应该被记录（"如何解决了 X"）
- **项目/需求评审结束后** — 评审决策和分歧需要归档
- **技术调研有结论时** — 调研结论（即使是否定结论）有价值
- **出错/踩坑后** — 教训是最宝贵的知识

**以下情况不应触发：**
- 纯执行性任务完成（没有新知识产生）
- 个人临时笔记（除非明确标记为有价值的）
- 已归档且未变更的内容
- 敏感/私密信息（不应进入共享知识库）

## Process

### Step 1: 扫描候选内容（Candidate Detection）

扫描以下位置寻找可归档内容：

| 来源 | 扫描方式 | 信号 |
|------|----------|------|
| `memory/YYYY-MM-DD.md` | 当日日志 | 新增的"决策"、"教训"、"问题"段落 |
| `reports/` 目录 | 新文件 | SKILL.md、分析报告、调研结果 |
| `agents/{role}/status.json` | `last_completed` + `last_output` | 新完成的任务产出 |
| 飞书文档 | `feishu-sync-log.json` | 新同步到飞书的文档 |
| 群聊消息 | 当日摘要 | 重要讨论和结论 |

**识别信号（满足任一即候选）：**
- 内容包含"决定"、"决策"、"方案"、"选择"
- 内容包含"教训"、"踩坑"、"问题"、"修复"、"不要"
- 内容包含"SOP"、"流程"、"规范"、"标准"
- 内容包含"为什么" + 结论（因果推理）
- 内容是 SKILL.md 文件
- 内容标记了 `@archive` 标签（手动标记）

### Step 2: 分类标签

为每个候选内容打 1-2 个分类标签：

| 主标签 | 子标签 | 说明 | 示例 |
|--------|--------|------|------|
| `🔧 技术` | 架构 | 系统设计决策 | "选择 LanceDB 而非 Pinecone 的原因" |
| `🔧 技术` | 工具 | 工具/库选型 | "飞书 API vs 腾讯文档 API 对比" |
| `🔧 技术` | 踩坑 | 技术问题与解决 | "飞书 API 429 限流处理策略" |
| `📋 产品` | 需求 | 需求分析与决策 | "日报双通道输出的设计理由" |
| `📋 产品` | 设计 | 产品设计原则 | "标题规范设计：类型+日期+摘要" |
| `📋 产品` | 竞品 | 竞品调研结论 | "addyosmani agent-skills 模板分析" |
| `🔄 流程` | SOP | 标准操作流程 | "Agent 任务交接协议" |
| `🔄 流程` | 规范 | 团队规范 | "status.json 字段规范" |
| `🔄 流程` | 仲裁 | 冲突解决规则 | "多 Agent 并发写入仲裁规则" |
| `💡 教训` | 错误 | 出错记录 | "忘记回写飞书链接导致找不到文档" |
| `💡 教训` | 改进 | 流程改进 | "subagent 返回格式标准化过程" |
| `💡 教训` | 反模式 | 不该做的事 | "不要盲写 Bitable 行（读-改-写原则）" |

### Step 3: 更新 MEMORY.md（长期记忆）

**MEMORY.md 追加格式：**

```markdown
### {YYYY-MM-DD} — {简短标题}

**分类:** {主标签} / {子标签}
**来源:** {task_id 或 文件路径}
**关联文档:** {飞书 URL 或本地路径}

**核心内容:**
{2-5 句话总结，包含：发生了什么、为什么重要、结论是什么}

**关键词:** #{tag1} #{tag2} #{tag3}
```

**追加位置：** MEMORY.md 末尾，按时间倒序。

**去重规则：**
- 如果已有相同来源（task_id 或 file_path）且内容未变更 → 不重复追加
- 如果同一主题有更新 → 更新已有条目，标注"更新于 {date}"

### Step 4: 更新 memory/YYYY-MM-DD.md（日常日志）

在当日日志末尾追加"知识归档"区块：

```markdown
## 📚 知识归档 (auto-generated by knowledge-base-update)

### 🔧 技术 / 工具
- [飞书 API 限流处理] 遇到 429 时等 Retry-After + 失败后写入 pending 队列
  - 来源: feishu-sync skill 设计 | task: custom-skills-build-001
  - 关联: reports/skills/feishu-sync/SKILL.md

### 📋 产品 / 设计
- [标题规范] 飞书文档标题统一为 [类型] + 日期 + 摘要
  - 来源: feishu-sync skill 设计 | task: custom-skills-build-001

### 🔄 流程 / SOP
- [Agent 任务交接] 规范了 task_id、blocker、output_path 的传递格式
  - 来源: agent-handoff skill 设计 | task: custom-skills-build-001

### 💡 教训
_(本日无)_

---
📊 本日归档: 3 条 | 总知识库条目: {N} 条
```

### Step 5: 飞书同步（可选）

对于高价值条目（标记为"重要"或涉及团队协作的），可选地同步到飞书知识空间。触发条件：
- 条目涉及 ≥2 个角色的协作
- 条目标记了 `@important` 或 `@team`
- 条目标签为 `🔄 流程/SOP`（团队规范必须共享）

同步方式：创建独立飞书文档或追加到"团队知识库"文档。

## Rationalizations

| 常见借口 | 反驳 |
|----------|------|
| "这个大家都知道，不用归档" | "大家都知道" = 最容易丢失的知识。现在团队 3 个人都"知道"，下周新成员加入时他就不知道。归档是为未来的团队成员做的。 |
| "我就记在脑子里了，不会忘" | 人的记忆不可靠。3 个月后你还能记住今天为什么选方案 A 而不是方案 B 吗？写下来。 |
| "归档太花时间了" | 每条归档 2-3 分钟。踩同一个坑的代价可能是 2-3 小时。ROI 是 60:1。 |
| "这个教训太丢人了，不好意思写" | "不要用 rm -rf 清理 workspace" —— 这条归档可能拯救下一个踩坑者。教训没有丢人的，只有有价值的。 |
| "MEMORY.md 越来越长，会不会太乱" | 定期整理（建议每月一次）合并相关条目。MEMORY.md 不是日志，是精选。用 `memory-hygiene` skill 清理。 |
| "我只是临时做了个决定，不算知识" | 决策 = 知识。为什么做这个决定？依据是什么？如果被推翻了，为什么？这些都是最有价值的知识。 |
| "分类标签太复杂了" | 4 个主标签、12 个子标签，覆盖 95% 的场景。如果找不到合适的，用"其他"然后 suggest 新标签。标签体系可以演进。 |
| "飞书同步是多余的" | 本地知识库给自己看，飞书知识库给团队看。两者不互斥。用 `@team` 标签标记需要共享的条目即可。 |

## Red Flags
- 连续 3 天无新归档条目 → 要么团队没产出（异常），要么归档阈值太高（调整识别信号）
- MEMORY.md 中同一来源出现多条重复条目 → 去重逻辑失效
- 归档条目缺少来源引用 → 不可追溯，降低可信度
- `memory/YYYY-MM-DD.md` 的知识归档区块超过 3 天未生成 → skill 可能未正常触发
- 分类为 `💡 教训` 的条目数量突然暴增 → 流程或工具存在系统性问题
- 条目标记了 `@team` 但未同步到飞书 → 团队知识共享链路断裂

## Verification
1. 完成一天的工作后，检查 `memory/YYYY-MM-DD.md` 是否有"知识归档"区块
2. 检查 MEMORY.md 是否有当天的归档条目，且格式正确
3. 随机抽查 3 条归档，验证"来源"是否可追溯（文件存在、链接有效）
4. 搜索 MEMORY.md 中的重复 task_id，确认无重复归档
5. 模拟一个包含"决策"和"教训"的日报，验证识别信号是否触发
6. 检查 `@team` 标记的条目是否全部同步到了飞书
