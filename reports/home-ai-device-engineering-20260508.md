**作者：** Stephen

## 1. 需求概述

Boss 计划购买 Mac 作为本地服务器，搭配一个物理前端设备，构成家庭 AI 管家系统。核心能力：

| 功能 | 描述 |
|------|------|
| 语音对话 | ASR → LLM → TTS 全链路 |
| 视频捕捉 | 实时人物 / 物品识别 |
| 家庭管家 | 家务安排 + 情绪陪伴 |

---

## 2. 技术栈选型

### 2.1 ASR（语音识别）

| 方案 | 优点 | 缺点 | 推荐场景 |
|------|------|------|----------|
| **whisper.cpp** (本地) | 离线可用，隐私保障，Mac M 系列 GPU 加速强 | 首次加载慢，模型文件大（≥ 140MB） | **首选**，主推 |
| whisper (Python/openai-whisper) | 精度高，生态成熟 | 需要 Python 环境，依赖多 | 备选 |
| 云端 API（Whisper API / 讯飞 / 阿里） | 精度好，无需本地算力 | 依赖网络，有隐私顾虑，延迟+往返耗时 | 离线 fallback |

**推荐决策：**
- 主用 **whisper.cpp**（C++，跨平台，Mac 通过 ML Compute 加速）
- 备用 **OpenAI Whisper API**（网络中断时自动切换）
- 建议模型：`base` 或 `small`（平衡速度与精度），`medium` 精度更高但慢 3-4x

### 2.2 LLM（大语言模型）

| 方案 | 优点 | 缺点 | 推荐场景 |
|------|------|------|----------|
| **Ollama + Llama 3/3.1/3.2** (本地) | 完全离线，隐私无忧，Mac M 系列 GPU 加速 | 响应速度依赖模型大小，上下文有限 | **主推** |
| Ollama + 其他开源模型 (Mistral, Qwen2.5) | 特定任务可能更强 | 需调优 | 备选 |
| 云端 API（GPT-4o / Claude / DeepSeek） | 智力强，响应快 | 依赖网络，有隐私顾虑 | 辅助复杂推理 |

**推荐决策：**
- 主用 **Ollama + Llama 3.2 8B**（Mac M 系 16GB+ RAM 可流畅运行）
- 复杂推理回退到云端 API
- Ollama 支持 API 兼容格式，切换成本低

### 2.3 TTS（语音合成）

| 方案 | 优点 | 缺点 | 推荐场景 |
|------|------|------|----------|
| **edge-tts** (微软在线) | 免费，质量好，多语言 | 依赖网络，非完全本地 | 首选高质量 |
| **piper-tts** (本地) | 完全离线，延迟低 | 声音自然度略逊 | 主推离线 |
| ElevenLabs | 质量最高，自然度高 | 收费，需 API Key | 追求拟真可选 |

**推荐决策：**
- 日常对话用 **edge-tts**（延迟低，质量接近真人）
- 离线需求用 **piper-tts**（支持中文，需要本地安装）
- 两者均可通过 Python 一行调用

### 2.4 计算机视觉

| 方案 | 优点 | 缺点 | 推荐场景 |
|------|------|------|----------|
| **YOLO (Ultralytics)** | 精度高，生态成熟，多目标实时 | 算力要求较高 | **主推目标检测** |
| **MediaPipe** | 延迟极低，专为边缘优化，人脸/手势/姿态 | 目标检测精度弱于 YOLO | 人脸识别 + 姿态 |
| **OpenCV** | 万金油，图像处理基础库 | 不直接做识别，需配合模型 | 预处理 + 辅助 |

**推荐决策：**
- 物体检测：**YOLOv8 nano**（`yolov8n.pt`，Mac CPU 可实时）
- 人脸检测：MediaPipe Face Detection（速度快）
- 组合：MediaPipe 做快速人脸初筛 → YOLO 做细粒度物品识别

### 2.5 唤醒词

| 方案 | 优点 | 缺点 | 推荐场景 |
|------|------|------|----------|
| **openWakeWord** | 开源，Python 生态，训练自定义词 | 需持续监听，CPU 占用 | **首选** |
| **Porcupine** | 商用级精度，超低功耗 | 需购买 license，自定义词有限制 | 追求稳定可选 |
| **snowboy** | 老牌，生态广 | 已停止维护 | 不推荐新项目 |

**推荐决策：** `openWakeWord`（开源，免费，支持自定义唤醒词训练）

## 3. 系统架构

### 3.1 整体架构

```
┌─────────────────────────────────────────────────────────┐
│                    物理前端设备                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │ 麦克风阵列 │  │  摄像头   │  │ 唤醒词引擎 │              │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘              │
│       │             │             │                     │
│       ▼             ▼             ▼                     │
│  ┌──────────────────────────────────┐                  │
│  │      前端处理（边缘计算）           │                  │
│  │  ASR(whisper.cpp) → 音频预处理    │                  │
│  │  视频流 → MediaPipe 人脸检测       │                  │
│  │  唤醒词检测 → VAD (WebRTC VAD)    │                  │
│  └──────────────┬───────────────────┘                  │
└─────────────────┼────────────────────────────────────────┘
                  │ LAN (WiFi, 建议 < 5ms 延迟)
                  ▼
┌─────────────────────────────────────────────────────────┐
│               本地 Mac 服务器                            │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │  Ollama  │  │  业务逻辑  │  │  数据存储  │              │
│  │  LLM 8B  │  │  Python   │  │ SQLite   │              │
│  └──────────┘  └──────────┘  └──────────┘              │
│                                                         │
│  API 网关 (FastAPI / Flask)                            │
└─────────────────────────────────────────────────────────┘
```

### 3.2 设备端 vs 服务器端分工

| 模块 | 位置 | 说明 |
|------|------|------|
| 唤醒词检测 | **前端设备** | 持续运行，低功耗，检测到唤醒词才唤醒 |
| VAD（语音活动检测） | **前端设备** | 判断说话开始/结束，过滤静音 |
| ASR（语音识别） | **前端设备** | whisper.cpp 本地运行，文本传给服务器 |
| LLM（语言模型） | **Mac 服务器** | Ollama 承载，8B 模型推理 |
| TTS（语音合成） | **前端设备** | edge-tts/piper-tts，本地播放 |
| 视频帧捕获 | **前端设备** | 摄像头持续采集，周期性推送关键帧 |
| CV 推理（物体/人脸） | **Mac 服务器** | YOLO/MediaPipe 在服务器 GPU/CPU 运行 |
| 业务逻辑 / 状态管理 | **Mac 服务器** | 家务安排、对话历史、自动化规则 |

### 3.3 网络要求

| 场景 | 需求 | 方案 |
|------|------|------|
| 正常对话 | WiFi LAN，延迟 < 10ms | 前端→Mac 服务器 RPC |
| 视频流 | 局域网带宽 ≥ 50Mbps | MJPEG 流或按需推送关键帧 |
| 离线 fallback | 无网络时降级为纯本地 | 前端设备自行承担 ASR+LLM+TTS（缩小模型） |
| **推荐** | **2.4GHz/5GHz 双频 WiFi，NAS 同网段** | Mac 和前端设备在同一个 LAN |

### 3.4 模块间通信方式

| 通信模式 | 适用场景 | 推荐 |
|----------|----------|------|
| **gRPC** | 结构化请求/响应，低延迟，双向流 | **推荐**，前端→服务器 |
| **WebSocket** | 持续对话，服务器推送 | 语音流媒体传输 |
| **MQTT** | IoT 设备，轻量 pub/sub | 设备状态上报（可选） |
| **REST API** | 简单请求，简单集成 | 管理接口 |

**推荐：`gRPC`（主）+ `WebSocket`（语音流）+ `MQTT`（设备状态监控，可选）**

## 4. 硬件兼容性

### 4.1 前端设备选型对比

| 设备 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **树莓派 5 (4GB/8GB)** | 便宜，生态大，功耗低，GPIO 可扩展 | 算力弱，视频推理勉强 | ⭐⭐⭐⭐ 性价比首选 |
| **Jetson Nano** | GPU 强，AI 推理优化，CUDA 支持 | 贵，功耗大，缺货 | ⭐⭐⭐ 视频推理优先 |
| **旧 iPad（iOS）** | 屏幕 + 麦克风 + 摄像头齐全 | 无法运行 Python 后端，只能做显示层 | ⭐⭐ 展示端 |
| **Mac Mini（服务器同体）** | 无需额外硬件，一体化 | 不能做"前端捕捉" | 仅做服务器 |

**推荐方案：**
- 最佳性价比：**树莓派 5 (8GB)**
- 追求视频 AI 性能：**Jetson Nano + 树莓派 5**（分工：Pi 做采集，Nano 做推理）
- 简化方案（旧设备优先）：**旧 Android 手机**（支持 USB 摄像头，Termux 运行 Python）

| 模组 | 分辨率 | 特点 | 推荐度 |

| **Raspberry Pi Camera v2** | 8MP | 官方支持，低延迟 | ⭐⭐⭐⭐（Pi 专用） |
| **Intel RealSense D435** | 1080p + 深度 | 深度感知，贵 | ⭐⭐⭐（物体定位优先） |
| **USB 麦克风阵列** | 4 麦阵列 | 远场拾音，降噪 | ⭐⭐⭐⭐ |

### 4.3 麦克风方案

| 方案 | 适用场景 | 推荐度 |
|------|----------|--------|
| **单麦（3.5mm / USB）** | 近场（< 1m），简单场景 | ⭐⭐⭐⭐ 入门 |
| **麦克风阵列（ReSpeaker 4 Mic）** | 远场（3-5m），全屋覆盖 | ⭐⭐⭐⭐⭐ 推荐 |
| **ReSpeaker USB 6+1** | 强降噪需求 | ⭐⭐⭐⭐ |

**推荐：ReSpeaker 4 Mic Array（树莓派兼容，环形布局，支持波束成形）**

## 5. 延迟分析

### 5.1 语音对话端到端延迟

```
用户说话 → VAD检测 → ASR识别 → 网络传输 → LLM推理 → TTS合成 → 语音播放
  ↓         ↓          ↓          ↓          ↓          ↓          ↓
 0ms      ~200ms    ~300ms     ~5ms      ~800ms    ~200ms     即时
（说话时长）（等待结束） （whisper.cpp）  （LAN）  （Llama 8B） （edge-tts）
```

**分段延迟估算：**

| 阶段 | 耗时 | 说明 |
|------|------|------|
| VAD 检测（说话结束判断） | ~200-400ms | WebRTC VAD，语音停止后触发 |
| ASR（whisper.cpp base 模型） | ~300-500ms | Mac M 系 GPU 加速 |
| 网络往返 | ~5-10ms | WiFi LAN，同网段 |
| LLM 推理（Llama 3.2 8B） | ~500-1500ms | 取决于上下文长度 |
| TTS 合成（edge-tts） | ~200-400ms | 含网络往返 |
| **总计** | **~1.5-3.5s** | 首次响应时间 |

**优化空间：**
- LLM 推理：流式输出（streaming），首 token 500ms 内可出
- ASR：使用 `tiny` 模型可压到 200ms
- TTS：piper-tts 本地可压到 100ms 内

### 5.2 视频识别实时性

| 场景 | 帧率需求 | 延迟 | 方案 |
|------|----------|------|------|
| 人物检测（在家状态） | ~5 fps 即可 | < 500ms/帧 | YOLOv8n，Mac CPU 可跑 |
| 人脸识别（谁回家了） | ~10 fps | < 300ms/帧 | MediaPipe Face Detection |
| 实时监控（异常检测） | ~15 fps | < 200ms/帧 | 需要 Jetson Nano 或 Mac GPU |
| 物品识别（找东西） | 按需触发 | < 2s/次 | YOLOv8n + 自然语言查询 |

**结论：**
- 日常人物在场检测（5fps）无压力
- 实时异常监控建议用 **Mac GPU**（Core ML 加速 YOLO）
- 树莓派 5 可承担 5-10fps 的 YOLOv8n

## 6. 开发工作量估算

### 6.1 模块化工作量

| 模块 | 技术栈 | 难度 | 预估工时 |
|------|--------|------|----------|
| 唤醒词引擎 | openWakeWord | 中 | 8h |
| VAD + 语音采集 | WebRTC VAD / silero-vad | 低 | 4h |
| ASR 本地推理 | whisper.cpp / pywhisper | 中 | 8h |
| LLM 服务器 | Ollama + FastAPI | 低 | 8h |
| TTS 合成 | edge-tts / piper-tts | 低 | 4h |
| 视频采集 + 流推送 | OpenCV / GStreamer | 中 | 12h |
| CV 推理（物体/人脸） | YOLOv8 + MediaPipe | 中高 | 16h |
| 前端-服务器通信 | gRPC / WebSocket | 中 | 16h |
| 业务逻辑（家务/情绪） | Python + SQLite | 中 | 24h |
| 系统集成 + 异常处理 | 全栈 | 高 | 24h |
| **合计** | | | **~124h** |

### 6.2 推荐开发顺序

```
Phase 1（基础能力，~40h）
  唤醒词 → ASR → LLM → TTS（语音对话闭环）

Phase 2（视频能力，~40h）
  摄像头 → 人脸检测 → 物品识别

Phase 3（业务逻辑，~30h）
  家务安排 → 情绪陪伴 → 持久化存储

Phase 4（集成优化，~20h）
  全流程联调 → 离线 fallback → 性能调优
```

### 6.3 关键技术风险

| 风险 | 影响 | 缓解方案 |
|------|------|----------|
| Whisper 本地推理太慢 | 语音响应 > 5s | 换 small 模型或加钱上 Mac GPU |
| 树莓派 WiFi 不稳定 | 视频卡顿 | 有线以太网回退，MQTT 心跳检测 |
| LLM 幻觉（家务安排出错） | 任务执行错误 | 规则引擎兜底，关键操作确认 |
| 隐私（视频数据外传） | 安全合规 | 纯本地，所有数据不出局域网 |

## 7. 总结与推荐方案

### 推荐技术栈（最优性价比）

| 组件 | 推荐方案 |
|------|----------|
| 服务器 | Mac Mini M4，24GB+ RAM |
| 前端设备 | **树莓派 5 (8GB)** |
| 摄像头 | Logitech C920 |
| 麦克风 | ReSpeaker 4 Mic Array |
| ASR | whisper.cpp (`base` 模型) |
| LLM | Ollama + Llama 3.2 8B |
| TTS | edge-tts（在线）+ piper-tts（离线） |
| CV | YOLOv8n + MediaPipe |
| 唤醒词 | openWakeWord |
| 通信 | gRPC（主）+ WebSocket（语音流） |

### 下一步行动建议

1. **先验证语音链路**：树莓派 + USB 麦克风跑通 whisper.cpp，确认延迟可接受
2. **Mac 先安装 Ollama**：确认 Llama 8B 在本地推理速度
3. **小步迭代**：先做纯语音对话，再加视频能力
4. **预留离线能力**：确保核心功能在网络断开时仍可用
