## 一、Agent-based QA 业界最佳实践

### 1.1 典型案例

| 案例 | 机制 | 启示 |
|------|------|------|
| **Anthropic Constitutional AI** | 模型输出先经"宪法规则"自检，违规内容被标记并要求重写 | QA 内嵌于生成流程，而非外部附加 |
| **OpenAI Critic Model (GPT-4 Reviewer)** | 用专用 smaller model 对大模型输出打分，低于阈值触发重做 | 便宜模型做质量把关，节省成本 |
| **Microsoft AutoGen** | 多 Agent 角色扮演，critic agent 对方案提出质疑，原作者回应修改 | 结构化的"提意见 → 答辩 → 修订"流程 |
| **Devin (Cognition AI) 的评审层** | 每个任务输出附带 self-verification step，自查代码正确性 | 机器自检不能完全替代人工，但能捕捉 60-70% 明显问题 |

### 1.2 工作流类型对比

| 维度 | 自动触发 | 手动触发 |
|------|----------|----------|
| **适用场景** | 每次产出都要审（文档、代码、PRD） | 高风险决策、发布前、上线前 |
| **优点** | 不遗漏，标准化 | 节省 token，聚焦关键节点 |
| **缺点** | token 消耗高，容易审美疲劳 | 可能遗漏 |

**建议：** 对关键产出手动触发，对常规产出心跳时自动跑抽样审查。

### 1.3 独立 vs 嵌入

| 方案 | 独立性 | 成本 | 适用场景 |
|------|--------|------|----------|
| **QA 内嵌于生产 agent**（加 prompt checklist） | 低 | 低 | 小团队，轻量场景 |
| **独立 QA session + 同一模型** | 中 | 中 | 需要结构化评审流程 |
| **独立 QA agent + 独立模型** | 高 | 高 | 高质量要求，多角色协作 |

**结论：** QA agent 必须有独立 system prompt，不与生产 agent 共享上下文，避免"自己审自己"的盲区。

## 二、多模型角色校核（Multi-Model Verification）

### 2.1 常见模式

**模式 A：强 + 便宜互补**
```
DeepSeek 出方案 → MiniMax 挑刺 → DeepSeek 修订
```
- 适合场景：代码架构、方案设计
- 成本：几乎零额外成本（MiniMax 是固定月费）

**模式 B：三角色交叉**
```
A模型出方案 → B模型打分 → C模型评审
```
- 适合场景：高风险决策、技术选型
- 当前团队可用：DeepSeek（主）+ MiniMax（挑刺）+ （可选 Claude/o1 作为第三视角）

**模式 C：一致性检验**
```
同一任务两个模型独立输出 → QA agent 比对差异点
```
- 适合场景：PRD 关键结论、架构决策、数据指标
- 差异点 = 潜在风险点

### 2.2 对现有包月模型成本的影响

MiniMax M2.7 是固定月费，调用成本为零边际成本。**建议将 MiniMax 定位为"挑刺者"角色**，主要负责：
- 审查 Stephen 的架构文档（找矛盾/疏漏）
- 审查 Thomas 的 PRD（补全维度）
- 审查 Lucia 的运营决策（逻辑自洽性）

DeepSeek 作为主方案输出者，承担生产角色。

### 3.3 文档/代码审查 Checklist 模板

#### PRD 审查 Checklist
- [ ] 目标用户是否明确？是否区分核心用户 & 边缘用户？
- [ ] KPI 定义是否可量化？可观测？
- [ ] 是否有明确的非目标（不做什么）？
- [ ] 边界情况（error state、empty state、网络异常）是否覆盖？
- [ ] 依赖方是否已确认（工程、设计、运营）？
- [ ] 数据埋点是否规划？
- [ ] 是否与现有产品逻辑冲突？

#### 架构文档审查 Checklist
- [ ] 选型理由是否有数据/场景支撑？（例：70B 模型需对应硬件规格）
- [ ] 单点故障风险是否识别？
- [ ] 扩展性：未来 3 个月scale up 是否要重写？
- [ ] 第三方依赖是否有备份方案？
- [ ] 成本估算是否列出（推理成本、运维成本）？

#### 运营/产品决策审查 Checklist
- [ ] 是否基于数据还是直觉？数据来源是什么？
- [ ] 决策的可逆性如何？（能撤回吗？）
- [ ] 最坏情况是什么？有止损方案吗？
- [ ] 利益相关方是否都已知会？

## 四、QA Agent 角色定义

### 4.1 名称候选

| 名称 | 语气 | 适合场景 |
|------|------|----------|
| **质检员** | 偏官方 | 正式流程嵌入 |
| **守门人** | 偏安全 | 强调上线 Gate 角色 |
| **审稿人** | 偏文化 | 文档/内容审查 |
| **校验官** | 偏技术 | 强调验证逻辑 |
| **挑刺师** | 偏口语/活泼 | 轻松指出问题 |

**推荐：** 内部称**「校验官」**，对外/文档中称**「守门人」**。

### 4.2 职责边界

```
审查范围：
├── 代码（Stephen 输出）      → 架构合理性、选型匹配、边界覆盖
├── 文档（PRD、技术方案）     → 维度完整性、逻辑自洽、可逆性
├── 产品决策（Lucia 输出）   → 数据支撑、风险识别、利益相关方
└── 上线前检查（小程序）      → 体验缺陷清单、埋点验证

不审查：
✗ 日常闲聊内容
✗ 纯创意/ brainstorming 阶段（过早审查扼杀创意）
✗ 紧急救火类临时决策（可补审，不阻塞）
```

### 4.3 触发机制

| 产出类型 | 触发方式 | 执行者 |
|----------|----------|--------|
| PRD 初稿 | 手动触发 | Thomas 提交后点「送审」 |
| 架构文档 | 手动触发 | Stephen 提交后点「送审」 |
| 产品决策 | 手动触发 | Lucia 发布前点「送审」 |
| 小程序上线 | 自动触发 | 发布前 Gate |
| 日常产出（日报、周报） | 心跳抽样 20% | QA 自动抽样 |

### 4.4 失败模式 & 防御

| 失败场景 | 防御机制 |
|----------|----------|
| QA 说"通过"但实际有问题 | Boss 保留最终否决权；QA 决策可申诉 |
| QA 过度保守，全部打回 | 设置通过率指标（参考值：50-70% 通过率健康） |
| QA 过度宽松，流于形式 | 定期抽检 QA 历史记录，Boss review QA 质量 |
| 生产 agent 绕过 QA | 关键节点强制 gate，未审内容不打 tag「已验证」 |

## 五、实施方案推荐

### 方案 A：轻量版（加 QA checklist prompt）

**做法：** 在 Lucia 的 system prompt 里加入标准化审查 checklist，每次心跳时自动跑。

**优点：** 零成本，立即可用  
**缺点：** QA 逻辑与生产逻辑混在一起，独立性和客观性不足

### 方案 B：独立 QA agent（独立 session + 独立模型）

**做法：** 创建独立 agent「校验官」，使用 MiniMax 模型（或 DeepSeek），Boss/Lucia 手动触发审查关键产出。

**优点：** 角色独立，审查逻辑清晰可迭代  
**缺点：** 需要维护独立 agent 配置

### 方案 C：双模型校验（DeepSeek + MiniMax 对比）

**做法：** 关键产出由两个模型分别出审查意见，QA agent 对比差异，聚焦争议点。

**优点：** 利用现有包月资源，零额外模型成本  
**缺点：** 两份审查意见增加 Boss 阅读成本，需要有聚合机制

### 推荐方案：**B + C 混合**（Phase 1 先走 B，Phase 2 升级到 B+C）

**理由：**
1. 当前最大痛点是**没有结构化 QA**，轻量版（方案 A）治标不治本
2. 独立 QA agent（方案 B）可以快速建立标准流程，2-3 天可落地
3. 双模型校验（方案 C）依赖 MiniMax 稳定性和 prompt 调优，Phase 2 再上
4. 微信小程序自动化测试（工具链存在但维护成本高）放在 Phase 2

**落地路径：**

```
Phase 1（本周~下周）
├── 创建独立 QA agent「校验官」
├── 配置 MiniMax 模型（固定月费，零边际成本）
├── 编写标准化审查 prompt（PRD/架构/决策三套 checklist）
├── 确定关键产出触发机制（手动送审）
└── 测试：先用 Thomas 的 PRD 跑一轮

Phase 2（2-4周后）
├── 加入 DeepSeek + MiniMax 双模型对比校验
├── 引入 ClawHub 上的 relevant skills（code-review、architecture-critic）
├── 小程序自动化测试 POC（miniprogram-automator）
└── 建立 QA 通过率监控指标
```
