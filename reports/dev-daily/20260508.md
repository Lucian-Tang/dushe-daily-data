## 二、流程优化灵感

### 灵感 1：Agent 执行"过程可见性"是下一个关键瓶颈

从 Stage CLI 和 SwiftWork 都可以看到一个共同信号：**当前 AI 编程工具的问题是"AI 做了很多，但人类看不见过程"。**

当下我们团队可能面临的问题：
- Agent 出了一份代码，但人类 review 时不知道它"为什么这么做"
- 变更范围超出预期，但直到合并才发现
- 多人协作时，AI 生成的内容没有可追溯的决策路径

**建议方向：** 在工作流中引入"AI 变更分步阅读"环节，让 Agent 的每一步执行都有可查看的记录。这不只是工具问题，是流程设计问题。

---

### 灵感 2："长时间计算任务"的调用模式值得标准化

从 HN 讨论中看到一个实际案例（关于"长时间计算请求"的处理方式）：不同语言、不同模型、不同交付形式（文件/数组）混在一起，团队缺乏统一的"长时任务提交-等待-回调"规范。

**建议方向：** 我们可以定义一个内部标准：
```
任务提交 → 唯一 ID → 状态轮询 / Webhook 回调 → 结果拉取
```
这套模式一旦标准化，AI Agent 和后台任务系统的对接成本会大幅降低。

## 三、技能 / 框架

### 技能 1：MCP（Model Context Protocol）—— Agent 的 API 标准

**核心观点：** "MCP is APIs for Agents"（dev.to 文章标题）

MCP 正在做的事情，和 REST API 当年做到的事情一模一样：让不同系统之间可以用统一协议通信。只不过 REST 是给人用的，MCP 是给 Agent 用的。

**对团队的潜在价值：**
- 我们正在搭建的 AI 中台，各个模块之间如果用 MCP 连接，AI 能自主发现和使用工具，而不是硬编码调用
- 新成员加入时，学习成本大幅降低——懂 MCP 就懂了整个工具生态

**建议：** 团队内部推进 MCP 协议普及，作为 AI 工具集成的首选方案。

### 技能 2：ProgramBench——LLM 程序重建能力的基准测试

- 来源：HN，115 points, 63 comments
- 链接：https://arxiv.org/abs/2605.03546

**对团队的启发：**

这个测试问了一个很实际的问题："LLM 能否从零重建一个现有程序？" 结果暴露了当前模型的盲区——对隐式依赖、上下文跨度大的任务，模型表现远不如预期。

对我们团队的意义：**不要高估 AI 的"代码迁移"能力**。我们做代码重构或跨项目迁移时，不能指望 AI 完全自主完成，人类的架构理解仍然不可替代。但这个测试也给了明确的方向：未来如果想让 AI 更可靠地完成这类任务，需要在"上下文保持"和"隐式依赖识别"上有突破。
