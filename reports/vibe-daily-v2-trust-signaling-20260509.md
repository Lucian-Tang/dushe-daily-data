# AI靠谱信息管家 V2.0 — 信任传达设计方案
**产品经理：Thomas**
**日期：2026-05-09**
**文档状态：初稿**

---

## 一、背景与核心问题

"今天毒什么"小程序已有毒舌点评风格积累了大量用户情感连接。V2.0 引入可信度评分和多源验证后，面临一个经典冷启动问题：**用户凭什么信你的评分？**

本方案不讨论评分的技术实现，只聚焦一个命题：**如何让用户在看到评分的第一秒就产生"这个平台有点东西"的信任感。**

---

## 二、信任设计策略

### 2.1 参考案例分析

#### Airbnb 的信任系统
- **透明房东档案**：历史评价、身份验证徽章、回复率、加入时间全部外显
- **"超赞房东"徽章**：不是数字评分，而是一个身份标签，降低认知负担
- **评价可见性**：双向评价且不可删除，但可隐藏负面评价（争议设计）
- **关键洞察**：Airbnb 从不只展示一个数字，而是展示"一个完整的人/房源的立体画像"

#### Wikipedia 的编辑机制
- **版本历史透明**：每次编辑可回溯，有"最后编辑者"和"编辑摘要"
- **"本文被引用X次"外显**：不是"可信度90%"，而是"有X个来源支撑"
- **讨论页机制**：争议内容有专门的讨论区，呈现分歧而不是掩盖
- **关键洞察**：Wiki 的信任不是来自"官方认证"，而是来自"过程透明"和"开放质疑"

#### 美团评分
- **评分分布图**：不只显示4.2分，还显示"5星占60%、1星占5%"的分布
- **"XX人说准"外显**：用同行背书替代自卖自夸
- **时间衰减提示**："该餐厅3个月前评分4.5，最新评分4.1"——主动提示变化
- **关键洞察**：中国用户对评分最敏感的不是分值本身，而是"这个分数有没有刷过"

### 2.2 可复用 UX 模式提炼

| 模式名称 | 核心机制 | 适用场景 | 在本产品的落地 |
|---------|---------|---------|--------------|
| **信任轨迹图** | 展示信息来源链条 | 信息有多源验证时 | 每条信息下方显示"已验证3个来源" |
| **徽章系统** | 可识别身份标签替代纯数字 | 建立长期信任 | "已核验来源""双盲评审中"等徽章 |
| **过程透明化** | 让用户看到评分怎么来的 | 消除黑箱猜疑 | 评分说明页完整开放 |
| **同行背书** | "XX人认为有用"替代自我宣传 | 社交流量转化 | "今天有127人查阅了该信息" |
| **主动示弱** | 主动标注不确定性 | 长期诚信人设 | "本条信息来自单一来源，建议交叉验证" |

## 三、评分方法论透明化：评分说明页面设计

### 3.1 信息架构（IA）

```
评分说明页（可从任一信息详情的"?"入口进入）
│
├── A. 什么是可信度评分
│   ├── 定义：我们的可信度评分衡量的是"信息被多个独立可信来源交叉验证的程度"
│   ├── 评分范围：1-5星，不采用百分制（降低强迫感）
│   └── 评分不意味"对错"，只意味"验证充分程度"
│
├── B. 评分是怎么算出来的
│   ├── 3个核心维度（雷达图形式展示）
│   │   ├── 来源多样性（1-5分）：有几个独立来源？
│   │   ├── 来源权威性（1-5分）：来源本身的可信度
│   │   └── 时间新鲜度（1-5分）：信息是否有时效性？
│   ├── 权重说明：3个维度各占1/3，不加权（简单可解释）
│   ├── 计算示例：[真实案例] "XX事件" → 3源×2权威×新 = 4星
│   └── 自动化 + 人工复核：AI初评 + 编辑团队抽检
│
├── C. 这个信息来源是谁
│   ├── 来源列表：每个来源可展开
│   │   ├── 来源名称 + logo
│   │   ├── 来源类型（官方媒体/自媒体/政府公开数据/学术论文...）
│   │   └── 该来源的历史准确率（如果积累了数据）
│   └── "如何判断来源可信"：给用户的科普小贴士
│
├── D. 评分有变动时会通知我吗
│   ├── 信息更新机制说明
│   ├── 重大变动（评分从4星→2星）会推送通知
│   └── 历史版本可查看
│
└── E. 我不相信这个评分，怎么反馈？
    ├── "我认为评分有误"入口（重要！让用户有出口）
    ├── 反馈处理承诺：48小时内回复
    └── 反馈可见化：其他用户可看到"XX人认为需要复核"
```

### 3.2 评分卡片的即时呈现（用户不点进详情页时看到什么）

```
┌─────────────────────────────────────────────┐
│  [毒舌风格引导语]                             │
│  "这消息靠不靠谱？我们验了，你看着办。"         │
│                                             │
│  ┌──────┐  ┌─────────────────────────────┐  │
│  │ ⭐⭐⭐⭐ │  可信度 4.2 / 5                 │  │
│  │  4星  │  [查看凭什么]  ← 可点击          │  │
│  └──────┘  └─────────────────────────────┘  │
│                                             │
│  ✓ 已验证来源：3个独立来源                    │
│  ✓ 包含1个权威媒体                            │
│  ⚠  信息发布于5天前，建议交叉确认             │
└─────────────────────────────────────────────┘
```

**设计原则**：
- 先给结论，再给解释（倒金字塔）
- 不用纯数字，用"星级 + 来源数量"双重信号
- 主动暴露不确定性（"建议交叉确认"），比隐藏更能建立信任

## 四、信任阶梯：用户成长路径

### 4.1 新用户（第1-7天）：建立"有点东西"的初印象

**核心目标**：不要急于让用户相信你的评分，先让他觉得"这个平台有点认真"。

| 触点 | 传递信息 | 设计手段 |
|-----|---------|---------|
| 首次看到评分 | "我们不是在瞎评分" | 强制展示评分依据（哪怕只有1个来源也展示） |
| 第一次点击详情 | "评分方法论是公开的" | 评分说明页用浅显语言，不是技术文档 |
| 看到第一条低分信息（2星） | "我们敢说这东西不太行" | 低分信息不隐藏，反而重点标注"⚠️ 待验证" |
| 第一次毒舌点评出现 | "风格一致，没有装正经" | 毒舌文案保持，信任建立在"有个性"而非"官方背书" |

**关键指标**：详情页点击率（衡量用户是否愿意深入了解评分）

### 4.2 活跃用户（第8-30天）：建立"你说得准"的认知

**核心目标**：让用户积累"这平台说准过X次"的成功体验。

| 触点 | 传递信息 | 设计手段 |
|-----|---------|---------|
| 信息被更新/评分变动 | "我们会跟踪验证，不是一次性评分" | 推送："之前你看的XX信息，我们更新了可信度评分" |
| 用户主动反馈被采纳 | "你帮我们变得更好" | 采纳反馈后给用户发"贡献者"徽章 |
| 积累查看历史 | "你是个认真的人" | 生成"你的验证记录"年度报告 |
| 社区互动（如果有） | "大家一起来验证" | 展示"XX人觉得这条需要复核"，同行背书 |

**关键指标**：评分变动提醒的打开率（衡量用户是否信任平台的持续追踪）

### 4.3 付费用户（第30天+）：建立"你是最靠谱的"壁垒

**核心目标**：付费不仅是去除广告，而是获得"更深度的信任服务"。

| 功能 | 传递信息 | 设计手段 |
|-----|---------|---------|
| 深度溯源报告 | "别人只能看到3层，你能看10层" | 付费解锁来源追溯链完整视图 |
| 优先人工复核 | "你说有问题，我们先看" | 付费用户反馈进入VIP队列 |
| 专属信任标签 | "你是经过验证的认真用户" | 付费用户发的反馈权重更高 |
| 错误预警订阅 | "我们会主动提醒你" | 订阅关键词 + 评分变动时主动推送 |
| 信任背书导出 | "你分享的信息有我们背书" | 付费用户分享的信息可附"已验证"徽章 |

**关键指标**：付费转化率、续费率（信任转化为商业价值）

## 五、反例设计：低可信度信息（2星）的呈现

### 5.1 设计目标

低分信息是信任体系的"试金石"：
- **做对了**：用户觉得"这平台敢说真话，敢认不知道"
- **做错了**：用户觉得"你否定了信息本身"，或引发法律风险（措辞不当）

### 5.2 具体呈现方案

#### 原则1：标注"不确定性"，不标注"错误"

❌ 错误写法："这条信息是假的"
✅ 正确写法："该信息目前仅有单一来源，且来源为自媒体，缺乏交叉验证"

❌ 错误写法："这条新闻是谣言"
✅ 正确写法："该事件尚无权威媒体跟进，事件真实性待确认"

**原因**："假/谣言"是定性，有法律风险；"缺乏验证/待确认"是陈述事实，无懈可击。

#### 原则2：给出行动建议，不只是评分

```
⚠️ 可信度 2/5 — 单源信息，建议谨慎

[该信息来自]
  • 自媒体平台（1篇）
  • 发布于 12 天前
  • 无官方回应或权威媒体跟进

[我们建议]
  • 先存疑，等更多报道
  • 如果你看到其他来源，欢迎 [帮我们验证]

[你也想知道]
  127人查阅了该信息，其中 34人认为需要进一步核实
```

#### 原则3：毒舌风格在低分时的正确打开方式

毒舌风格不是"阴阳怪气"，而是"一针见血 + 给出口"：

❌ 阴阳怪气："哟，这条你也信？"
✅ 一针见血："这条目前只有一个人在说，我们建议你等更多人说了再下结论。——我们也在盯着，有新消息会告诉你。"

❌ 自以为是："这条是假的"
✅ 诚实示弱："说实话，单一来源的信息我们也拿不准。但你可以看看我们找到的其他线索。"

## 六、品牌调性融合：毒舌 + 严肃可信度

### 6.1 冲突分析

毒舌风格和可信度评分表面上有矛盾：
- 毒舌 = 随意、调侃、民间
- 可信度 = 严肃、专业、权威

但两者并不天然冲突，关键在于**毒舌指向谁**：

| 毒舌指向 | 效果 | 示例 |
|---------|-----|------|
| **指向信息本身**（调侃内容） | ✓ 一致 | "这消息传得比谣言还离谱——我们核实了，假的" |
| **指向行业乱象** | ✓ 一致 | "市面上90%的XX都是智商税，我们来验" |
| **指向平台自身**（自嘲） | ✓ 一致 | "我们也知道4星评分看起来很敷衍，所以做了这个页面给你看" |
| **指向低分用户的判断** | ✗ 冲突 | "你居然信这个？"（自毁人设） |

### 6.2 品牌语言融合指南

**核心公式**：毒舌的外壳 + 诚实的内核

```
外层（毒舌）：
- 语气轻松，敢怼，敢吐槽
- 用"我们""咱们"，不用"本平台""本公司"
- 适当用 emoji 和口语化表达

内层（可信度）：
- 事实陈述严谨（来源、数量、时间）
- 主动暴露不确定性
- 承认"我们也可能出错"（留反馈入口）

融合示例：
---
"网上铺天盖地都在说XX，我们去翻了翻——
好家伙，3个自媒体互相引用，零权威媒体跟进。
4星给不了，给个2星意思意思。
——等有实锤了再来更新。"
---
```

### 6.3 品牌语言红线

| ✗ 不要这样说 | ✓ 改成这样说 |
|------------|------------|
| "这条是假新闻" | "该信息缺乏多源验证" |
| "发布者是个骗子" | "该来源历史上准确率为X%" |
| "你被骗了" | "建议结合更多来源判断" |
| "我们保证100%准确" | "我们尽力核实，但请也保持独立判断" |

## 七、执行优先级建议

### P0（必须做，影响信任根基）
1. 评分说明页的信息架构落地（透明化是信任第一步）
2. 低分信息的标准化呈现（法律风险最高的场景）
3. 评分卡片不只显示数字，显示来源数量

### P1（做完 P0 后做，提升信任转化）
4. 评分变动推送通知（新信息的持续追踪能力）
5. 用户反馈入口（"你不信？告诉我们"）

### P2（增长阶段做，信任转化为商业）
6. 付费用户的深度溯源报告
7. 用户贡献者徽章体系
8. 信任背书导出功能
