# startup-ai-meeting-clerk-018 — 技术可行性报告

## 项目概述
会议纪要AI自动生成，支持录音转写+说话人识别+任务项提取+自动发送，Whisper+LLM技术方案。

---

## 1. 核心技术方案

### 技术栈
- **编程语言：** Python 3.11+
- **语音识别：** Whisper（OpenAI开源，本地部署，支持中文）/ Deepgram（云端备选）
- **说话人识别：** pyannote-segmentation（开源说话人 diarization）+ Resemblyzer（声纹聚类）
- **LLM处理：** GPT-4o / Claude 3.5（会议纪要生成+任务提取+摘要）
- **实时处理：** WebSocket（实时音频流）+ asyncio并发
- **存储：** SQLite/PostgreSQL（会议记录持久化）

### 系统架构
```
[音频输入] 
    → [Whisper实时转写] 
    → [pyannote说话人分离] 
    → [LLM纪要生成] 
    → [任务项提取] 
    → [发送/存储]
   Zoom/Teams/飞书/腾讯会议
```

### 核心模块
1. **会议平台接入** — Zoom/Teams/飞书/腾讯会议的API接入（Webhook实时音频流或录制文件拉取）
2. **语音转写引擎** — Whisper（本地推理，支持16kHz音频，中文识别准确率>95%）
3. **说话人分离器** — pyannote-segmentation（VAD+说话人切换检测），输出带时间戳的文本+说话人标签
4. **LLM纪要生成器** — 输入：转写文本+说话人标签，输出结构化纪要（时间/议题/结论/行动项）
5. **任务提取器** — 从纪要中提取「待办+负责人+截止时间」，输出任务卡片
6. **分发器** — 支持飞书/邮件/Notion/Confluence等平台发送

## 2. 技术可行性评估

**评分：4/5**

**理由：**
- ✅ Whisper成熟开源，中文语音识别效果优秀（<3%字错误率，Whisper Large-v3）
- ✅ pyannote说话人分离开源， diarization 效果在业界领先
- ✅ LLM已能很好地将转写文本结构化为会议纪要（GPT-4o/Claude均验证过）
- ✅ 会议平台API（Zoom/飞书）开放程度高，接入成本中等
- ⚠️ 实时会议纪要（Live）需要低延迟，架构复杂度更高
- ⚠️ 多人长时间会议（>1小时）的转写质量可能下降（说话人标签漂移）

## 4. 最佳实践/竞品技术方案调研

### 竞品分析
| 竞品 | 技术方案 | 特点 |
|------|---------|------|
| **Otter.ai** | 自研ASR+LLM | 最成熟，Zoom深度集成，$16/月 |
| **Fireflies.ai** | 多平台集成+任务提取 | AI任务提取是卖点，$18/人/月 |
| **Fathom** | Zoom插件+实时摘要 | 主打实时，低价策略 |
| **Otter AI Copilot** | 微软Teams集成 | 与Teams 365深度集成 |
| **飞书妙记** | 自研ASR+飞书生态 | 中文体验好，免费版有限 |

### 最佳实践
1. **飞书优先**：飞书会议在中国企业市场占率高，且API开放，先做飞书再扩展其他
2. **异步为主，实时为辅**：MVP做录制文件处理（延迟低，风险低），V1再上实时
3. **任务项提取是核心价值**：比纯纪要更有价值，需要LLM具备任务识别能力
4. **隐私合规**：企业会议数据敏感，需支持私有化部署，数据不经过第三方服务器

## 5. 技术风险与应对方案

| 风险 | 级别 | 应对方案 |
|------|------|---------|
| 说话人分离在大会议室（>8人）效果差 | 高 | 提供「手动标注说话人」功能作为兜底，设置人数上限提示 |
| 实时会议纪要延迟（>2分钟） | 中 | 流式Whisper推理（faster-whisper），优化管线延迟，目标<90秒 |
| 企业隐私合规 | 中 | 支持私有化部署（Docker一键），数据不出企业网络 |
| 会议平台API不稳定（飞书接口变更） | 低 | 版本隔离，接口异常监控+降级策略 |
