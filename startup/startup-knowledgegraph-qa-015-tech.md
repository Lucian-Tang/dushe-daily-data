# startup-knowledgegraph-qa-015 — 技术可行性报告

## 项目概述
企业知识库问答系统，基于知识图谱+LLM+RAG实现自然语言查询，打通企业内部文档孤岛。

---

## 1. 核心技术方案

### 技术栈
- **编程语言：** Python 3.11+
- **知识图谱：** Neo4j（主流图数据库）/ Amazon Neptune
- **LLM：** GPT-4o / Claude 3.5（自然语言理解+图查询生成）
- **RAG框架：** LangChain / LlamaIndex（文档解析+向量检索）
- **向量数据库：** Qdrant / Chroma（轻量） / pgvector（PostgreSQL扩展）
- **文档解析：** Unstructured.io / MarkItDown（PDF/Word/PPT提取）

### 系统架构
```
[用户问题] 
    → [意图识别器] 
    → [图谱检索（Neo4j Cypher） + RAG（向量检索）] 
    → [上下文融合] 
    → [LLM答案生成] 
    → [答案+溯源]
```

### 核心模块
1. **文档解析与入库管线** — PDF/Word/PPT → 文本提取 → Chunk分段 → 向量入库 + 图谱实体抽取
2. **意图识别器** — 分类问题类型（事实查询/关系推理/文档检索），决定路由策略
3. **知识图谱查询引擎** — 将自然语言转为Cypher查询，执行图数据库
4. **RAG检索器** — 向量相似度检索，Top-K相关文档块
5. **上下文融合器** — 合并图谱查询结果与RAG结果，去重+排序
6. **LLM答案生成器** — 输入：融合上下文+问题，输出：带溯源标注的答案

## 2. 技术可行性评估

**评分：4/5**

**理由：**
- ✅ RAG技术已非常成熟，LangChain/LlamaIndex生态完善
- ✅ Neo4j+Cypher在图数据库领域是标准，图查询与自然语言映射有成熟方案
- ✅ LLM已能很好理解图结构查询意图（Cypher生成Prompt优化后可做到85%+准确率）
- ✅ 企业知识库问答是明确痛点，市场需求真实
- ⚠️ 知识图谱构建成本高（需要大量人工或半自动标注）
- ⚠️ 图谱与RAG融合的路由策略需要精细调优

## 4. 最佳实践/竞品技术方案调研

### 竞品分析
| 竞品 | 技术方案 | 特点 |
|------|---------|------|
| **Notion AI Q&A** | RAG为主，简单KB | 面向C端，知识库有限 |
| **Glean** | 企业级搜索+RAG，连接所有SaaS | 产品成熟，$15k+/年，大型企业 |
| **Hebbia** | 深度RAG+LLM推理 | 主打金融/法律，B2B |
| **Yona** | 图增强RAG | 概念验证阶段 |
| **Neo4j + LangChain** | 开源方案组合 | 需要自己搭建，定制性强 |

### 最佳实践
1. **图谱分层建设**：先做文档级RAG，再逐步构建实体关系图谱（按使用频率优先）
2. **Cypher生成优化**：Few-shot Prompt，提供5-10个问答对作为示例，稳定生成准确率
3. **混合检索**：图谱查询结果与RAG结果做RRF（Reciprocal Rank Fusion）融合
4. **溯源设计**：每个答案必须附带「来源：文档名+页码/图谱关系路径」，不可出现无溯源答案

## 5. 技术风险与应对方案

| 风险 | 级别 | 应对方案 |
|------|------|---------|
| 图谱构建成本高，企业不愿投入 | 高 | 提供「零图谱」纯RAG模式，图谱作为可选增值层 |
| Cypher查询生成错误率 | 中 | 添加「查询验证」步骤，错误查询自动重写 |
| 企业数据安全（私有化部署） | 中 | 支持Docker一键部署，数据不出企业网络 |
| 多格式文档解析质量差 | 中 | 针对PDF扫描件、表格类文档专项优化 |

## 6. 落地步骤

1. **Week 1-3：** 文档解析管线（MarkItDown/Unstructured），RAG基础搭建，Qdrant部署

3. **Week 7-9：** 融合检索模块，RRF排序，LLM答案生成器，溯源标注
4. **Week 10-12：** 多格式支持（PDF/Word），企业SSO，Web UI（溯源展示）
5. **Week 13-14：** 灰度测试（2-3家企业），收集Bad Case，持续优化
